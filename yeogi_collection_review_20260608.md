# 여기어때 수집 실패 검토 메모

## 현재 실패 원인

- 현재 여기어때 검색 요청은 `goodchoice.kr`에서 `yeogi.com/domestic-accommodations`로 이동한 뒤 HTTP 403으로 차단된다.
- 실패 유형은 검색어 없음, 파서 오류, UI 오류가 아니라 Cloudflare/WAF 차단이다.
- Node.js 직접 요청은 브라우저에서 실행되는 JS 검증, 쿠키, 브라우저 지문, 세션 검증을 통과하지 못한다.

## 수집 가능 방향

1. 사용자 브라우저 세션 기반 수집
   - 단기적으로 가장 현실적인 방식이다.
   - 사용자가 정상 접속 가능한 Chrome/브라우저에서 검색 결과가 열린 뒤 DOM을 읽는 방식이다.
   - CAPTCHA, 로그인, 차단 안내 등은 사용자가 직접 처리해야 한다.
   - 사이트 정책을 지키는 범위에서 화면에 표시된 결과를 기록하는 보조 방식으로 검토한다.

2. 수동 CSV/HTML 가져오기
   - 여기어때 결과를 사람이 열어 확인한 뒤 HTML 저장본, CSV, 복사 붙여넣기 데이터를 앱에 넣는 방식이다.
   - 자동성은 낮지만 차단 리스크가 가장 낮고 검증용으로 적합하다.

3. 스크린샷/OCR 보조
   - 빠른 현황 파악용으로는 가능하지만 업체명, 가격, 광고 여부 오인식 가능성이 있어 최종 데이터로는 부적합하다.

4. 대체 채널 유지
   - 여기어때가 차단될 때는 떠나요/ONDA, 야놀자/NOL, 네이버를 기본 수집 축으로 유지한다.
   - 여기어때는 상태를 `실패/차단`으로 남기고 원인과 대안을 같이 표시한다.

5. 공식/제휴 API 확인
   - 가장 안정적인 방식이지만 영세업체나 초기 스타트업이 바로 확보하기는 어렵다.
   - 단기 실행 전략이 아니라 장기 제휴/영업 옵션으로 둔다.

## 권장 결론

- 단기: 현재 앱에는 여기어때 실패 원인과 수집 방향을 명확히 표시한다.
- 중기: 브라우저 세션 기반 확인 기능과 수동 가져오기 기능을 별도 옵션으로 만든다.
- 장기: 서비스가 커진 뒤 공식/제휴 API 확보 여부를 확인해 조건형 검색 데이터까지 안정적으로 수집한다.
