# 전 채널 예약가능률 수집 가능성 검토

## 핵심 정의

- 플랫폼이 실제 예약 전환율이나 예약 건수를 공개하지 않으면 `예약율`은 직접 수집할 수 없다.
- 현실적으로 앱에 넣을 지표명은 `예약 가능률` 또는 `판매 가능률`이 맞다.
- 기준식은 `예약 가능한 객실 또는 상품 수 / 확인 가능한 전체 객실 또는 상품 수`로 둔다.

## 네이버 기준 확정

네이버는 예약 화면 구조가 두 가지다.

먼저 상품 성격을 분리한다.

- `ACCOMMODATION_NIGHT`: 숙박 상품. 1박 예약가능률 계산 대상이다.
- `ACCOMMODATION_DAY_USE`: 데이유즈/당일 이용 상품. 1박 예약가능률에서는 제외하고 별도 카운트로 표시한다.
- 미분류 상품은 이름에 `데이유즈`, `대실`, `당일`이 있으면 데이유즈로 보고, 아니면 보수적으로 숙박 후보로 둔다.

1. `객실별 예약리스트`
   - 날짜 조건을 넣은 뒤 예약 가능한 객실 상품 수를 센다.
   - 산식: `예약가능 객실 상품 수 / 노출된 객실 상품 수`
   - 이때 `stock`이 0인 객실 상품도 분모에 포함한다. 이미 마감된 객실이기 때문이다.

2. `객실 종류별 리스트`
   - 객실 종류별 날짜 재고를 확인한다.
   - 산식: `sum(stock - bookingCount - occupiedBookingCount) / sum(stock)`
   - 검증 예시: 월명 글램핑 네이버예약 사업자ID `244049`, 체크인 `2026-06-14`, 성인 2명 기준 `16개 중 13개 가능`, 예약가능률 `81.3%`.

수집 흐름은 `네이버 플레이스 place_id -> 네이버예약 bookingBusinessId -> 예약 GraphQL searchBizItem -> daily schedule` 순서다.

## 채널별 판단

| 채널 | 수집 가능성 | 권장 지표 | 판단 |
|---|---|---|---|
| 네이버 | 가능 | 객실 단위 예약가능률 | bookingBusinessId가 있으면 날짜별 `stock`, `bookingCount` 확인 가능 |
| 야놀자/NOL | 부분 가능 | 날짜 조건 검색 결과 판매가능률 | 가격 노출/매진 신호 중심, 전체 객실 재고는 아님 |
| 여기어때 | 직접 자동수집 어려움 | 브라우저/수동 보조 지표 | Cloudflare 403. 사용자 정상 세션 DOM 또는 CSV/HTML 가져오기 필요 |
| 떠나요/ONDA | 가능 | 객실 단위 예약가능률 | 예약 달력 `detailDtos.isReservable` 기준 |

## 앱 적용 원칙

- 화면 표기는 `예약율` 대신 `예약가능률`로 한다.
- 네이버와 떠나요/ONDA는 객실 단위 지표로 우선 반영한다.
- 야놀자/NOL은 검색 결과 기준 판매가능 신호로 보조 반영한다.
- 여기어때는 브라우저 세션 기반 DOM 추출 또는 수동 가져오기 방식으로 보완한다.
- 전체 점수 산정 시 채널별 신뢰도를 분리한다.
